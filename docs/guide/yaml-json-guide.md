# anxwritter — YAML / JSON Guide

This guide is for analysts and data engineers driving anxwritter from
**YAML or JSON files**, not from Python. Every concept in the
[Python analyst guide](analyst-guide.md) is reachable from a YAML/JSON
file — this page maps each one to its dict shape and points you back at the
[Python reference](../reference/) for full field semantics.

If you only ever build charts from `.yaml` or `.json` inputs (typical when
generating data from spreadsheets, ETL pipelines, or other tools), you don't
need to read any Python code. Skip back to the reference docs only when you
hit a field you don't recognise.

---

## When to use YAML / JSON vs Python

| Use case | Recommended path |
|----------|------------------|
| Hand-edited investigation files | YAML (comments + readability) |
| Generated by an ETL pipeline | JSON (mechanical, no quoting surprises) |
| Layered org-wide defaults | YAML config files, applied via `--config` |
| Programmatic chart construction (loops, conditionals, DB cursors) | Python ([analyst guide](analyst-guide.md)) |
| One-off ad-hoc charts | Either — pick what your team writes faster |

YAML and JSON go through the same loader (`ANXChart.from_dict`), so anything
expressible in one is expressible in the other.

## Quick start

### YAML

```yaml
entities:
  icons:
    - id: Alice
      type: Person
      color: Blue
    - id: Bob
      type: Person
      color: Red

links:
  - from_id: Alice
    to_id: Bob
    type: Call
    arrow: '->'
    date: '2024-01-15'
```

Build it:

```bash
anxwritter chart.yaml -o output/chart.anx
```

### JSON

```json
{
  "entities": {
    "icons": [
      {"id": "Alice", "type": "Person", "color": "Blue"},
      {"id": "Bob", "type": "Person", "color": "Red"}
    ]
  },
  "links": [
    {"from_id": "Alice", "to_id": "Bob", "type": "Call",
     "arrow": "->", "date": "2024-01-15"}
  ]
}
```

```bash
anxwritter chart.json -o output/chart.anx
```

See the [CLI reference](../reference/cli.md) for all flags.

## Top-level keys

Every YAML or JSON file is a dict with up to 16 top-level keys. All are optional.

| Key | Purpose | Python reference |
|-----|---------|------------------|
| `settings` | Chart-wide configuration (background, layout, summary, …) | [settings.md](../reference/settings.md) |
| `entities` | The chart items, grouped by representation type | [entities.md](../reference/entities.md) |
| `links` | Connections between entities | [links.md](../reference/links.md) |
| `loose_cards` | Cards routed to entities/links by id at build time | [cards-and-grades.md § Loose cards](../reference/cards-and-grades.md#loose-cards) |
| `entity_types` | Pre-registered entity types (icon, color) | [visual-styling.md § EntityType](../reference/visual-styling.md#entitytype) |
| `link_types` | Pre-registered link types (color) | [visual-styling.md § LinkType](../reference/visual-styling.md#linktype) |
| `attribute_classes` | Display config for named attributes | [attributes.md](../reference/attributes.md) |
| `strengths` | Named line dash styles (`Confirmed`, `Tentative`, …) | [visual-styling.md § Strengths](../reference/visual-styling.md#strengths) |
| `datetime_formats` | Named date/time display formats | [datetime-and-timezones.md § DateTimeFormat](../reference/datetime-and-timezones.md#datetimeformat) |
| `palettes` | Custom palettes for ANB's "Insert from Palette" panel | [visual-styling.md § Palettes](../reference/visual-styling.md#palettes) |
| `grades_one`, `grades_two`, `grades_three` | Grade label collections | [cards-and-grades.md § Grade collections](../reference/cards-and-grades.md#grade-collections) |
| `source_types` | Source type dropdown labels | [cards-and-grades.md § Source types](../reference/cards-and-grades.md#source-types) |
| `legend_items` | Custom legend rows | [visual-styling.md § Legend items](../reference/visual-styling.md#legend-items) |
| `semantic_entities`, `semantic_links`, `semantic_properties` | i2 Semantic Library types | [semantic-types.md](../reference/semantic-types.md) |

A **config file** is a file containing only the configuration sections (no
`entities` or `links`). See the [config file section](#config-files) below.

## Entities

`entities` is a dict keyed by representation type. Each value is a list of
dicts; each dict's keys are the fields documented in [entities.md](../reference/entities.md).

| YAML key | Entity class | Best for |
|----------|--------------|----------|
| `icons` | Icon | People, vehicles, phones, objects |
| `boxes` | Box | Locations, accounts, organisations |
| `circles` | Circle | Events, meetings, incidents |
| `theme_lines` | ThemeLine | Timeline bands |
| `event_frames` | EventFrame | Time-bounded regions |
| `text_blocks` | TextBlock | Free-standing text |
| `labels` | Label | Transparent text overlays |

```yaml
entities:
  icons:
    - id: alice
      type: Person
      label: Alice Smith              # defaults to id
      color: Blue                     # icon shading
      date: '2024-01-15'              # quote it — see YAML gotchas below
      time: '14:30:00'
      attributes:
        Phone: '555-0001'
        Age: 39
        Active: true
  boxes:
    - id: hq
      type: Location
      width: 150
      height: 100
      bg_color: 'Light Grey'
  circles:
    - id: meeting_01
      type: Event
      diameter: 80
  theme_lines:
    - id: timeline
      type: Timeline
  event_frames:
    - id: surveillance
      type: Event
  text_blocks:
    - id: note_01
      type: Note
      label: 'Analyst note: cross-reference with case file 2024-007'
  labels:
    - id: title
      type: Label
      label: 'Operation Sunshine'
```

For the full per-class field list (defaults, type-specific fields like
`autosize` on Circle or `depth` on Box), see [entities.md](../reference/entities.md).

### Nested fields: `label_font`, `show`, `frame`

These accept inline maps in YAML/JSON exactly the way the Python `Font`,
`Show`, and `Frame` dataclasses do. The loader converts the dict to the
right dataclass automatically.

```yaml
entities:
  icons:
    - id: alice
      type: Person
      label_font:
        color: Red
        bold: true
        size: 12
      show:
        description: true
        date: true
        label: true
      frame:
        color: 16764057
        margin: 2
        visible: true
```

Field details: [Font](../reference/settings.md#font----chart-level-default-font),
[Show](../reference/entities.md#show-dataclass),
[Frame](../reference/entities.md#frame-dataclass).

`frame` is silently dropped on entity types that don't accept it (Box, Circle,
TextBlock, Label) and on links — so you can keep a uniform shape across a
generated dataset without per-row branches.

## Links

`links` is a flat list. Each item references entities by `id` via `from_id`
and `to_id`.

```yaml
links:
  - from_id: alice
    to_id: bob
    type: Call
    arrow: '->'                       # quote — '>' is YAML special
    label: 'Mobile call'
    date: '2024-01-15'
    time: '14:30:00'
    description: 'Intercepted, 2 minutes'
    strength: Confirmed
    line_color: '#FF0000'
    line_width: 2
    attributes:
      Duration: 120
      Call Type: Mobile
```

Arrow values: `'head'`/`'->'`, `'tail'`/`'<-'`, `'both'`/`'<->'`, or omit
for a plain line. Always quote — `>` and `<` are YAML special characters.

For full link field details (offset, multiplicity, fan_out, theme_wiring,
ordered, datetime_format, semantic_type, …) see [links.md](../reference/links.md).

### Multi-link rendering

```yaml
links:
  - from_id: A
    to_id: B
    type: Call
    multiplicity: single              # collapses parallel links into one + card stack
    fan_out: 50
    theme_wiring: keep_event
```

Valid string values: `multiplicity` ∈ `{multiple, single, directed}`;
`theme_wiring` ∈ `{keep_event, return_theme, next_event, no_diversion}`.

### Loose card attachment

To attach loose cards to a link by id, set `link_id` on the link first
(internal — not written to XML):

```yaml
links:
  - from_id: alice
    to_id: bob
    type: Call
    link_id: call_001                 # internal handle for loose cards
```

Then attach cards either inline via `cards:` on the entity/link, or
through the top-level `loose_cards:` section (see below) — useful when
cards come from a different data source than entities/links.

## Cards (inline)

Inline cards live under an entity's or link's `cards` list. The dict keys
match the [Card fields](../reference/cards-and-grades.md#card-fields).

```yaml
entities:
  icons:
    - id: alice
      type: Person
      cards:
        - summary: 'Primary suspect'
          date: '2024-01-15'
          time: '14:30:00'
          description: 'Identified from CCTV'
          source_ref: 'REP-001'
          source_type: Witness
          grade_one: 0                # 0-based index into grades_one.items
          grade_two: 'Probably true'  # ...or the registered grade name string
          timezone:
            id: 1                     # ANB UniqueID; 1 = UTC
            name: UTC

links:
  - from_id: alice
    to_id: bob
    type: Call
    cards:
      - summary: 'Intercept transcript'
        date: '2024-01-17'
        source_ref: 'RPT-001'
```

`timezone` requires both `date` and `time` to be set, otherwise ANB
silently ignores it (validation flags this with `timezone_without_datetime`).

For the full TimeZone UniqueID table (1–122) see
[datetime-and-timezones.md § Full timezone table](../reference/datetime-and-timezones.md#full-timezone-table-ids-1--122).

## Loose cards (top-level `loose_cards`)

When card data comes from a different source than entities (e.g. a
separate CSV or a different DB query in your ETL pipeline), use the
top-level `loose_cards:` key. Each entry routes to its target by
`entity_id` (matching an entity's `id`) or by `link_id` (matching a
link's `link_id`).

```yaml
entities:
  icons:
    - id: alice
      type: Person

links:
  - from_id: alice
    to_id: bob
    type: Call
    link_id: call_001               # internal handle for loose card routing

loose_cards:
  - entity_id: alice
    summary: 'New intel from interview'
    date: '2024-03-01'
    source_ref: 'INT-005'
    grade_one: 1

  - link_id: call_001
    summary: 'Wiretap transcript'
    date: '2024-01-17'
    source_ref: 'RPT-001'
```

Loose cards accept all the same fields as inline cards. Routing failures
(unknown `entity_id` / `link_id`) are caught at validation time as
`missing_target` errors. See
[cards-and-grades.md § Loose cards](../reference/cards-and-grades.md#loose-cards)
for the full Card field list.

## Attributes

Attributes are a flat key/value dict on each entity or link. The data type
is inferred from the YAML/JSON value:

| YAML/JSON value | Inferred type | Example |
|-----------------|---------------|---------|
| string | Text | `Phone: '555-0001'` |
| integer | Number | `Age: 39` |
| float | Number | `Balance: 12500.50` |
| boolean | Flag | `Active: true` |

```yaml
entities:
  icons:
    - id: alice
      type: Person
      attributes:
        Phone: '555-0001'
        Age: 39
        Balance: 12500.50
        Active: true
```

**Type consistency** — all uses of the same attribute name across the chart
must resolve to the same Python type. Mixing `Balance: 12500.50` (float) and
`Balance: unknown` (string) elsewhere triggers a `type_conflict` validation
error.

### Datetime attribute values

YAML's native datetime parser turns unquoted ISO timestamps into Python
`datetime` objects, which anxwritter recognises and serialises correctly:

```yaml
entities:
  icons:
    - id: alice
      type: Person
      attributes:
        Joined: 2024-01-15T10:00:00   # YAML parses this as a datetime
```

To keep a value as a string instead, quote it (`'2024-01-15'`).

### Configuring attribute display — `attribute_classes`

Each entry in `attribute_classes` requires a `type` (`text`, `number`,
`flag`, or `datetime`). Validation rejects any entry that omits it.

```yaml
attribute_classes:
  - name: Balance
    type: number
    prefix: '$'
    decimal_places: 2
    icon_file: cash

  - name: Phone
    type: text
    prefix: 'Tel: '
    icon_file: phone

  - name: Active
    type: flag
    show_if_set: true

  - name: Notes
    type: text
    merge_behaviour: add_line_break
    paste_behaviour: add_line_break
    font:
      bold: true
      color: Red
```

Full field reference + valid `merge_behaviour` / `paste_behaviour` values per
type: [attributes.md](../reference/attributes.md).

## Grades, source types, strengths

These sections set up dropdown options that entities/links/cards reference
by index or name (grades) or by name (source types, strengths).

### Grades

```yaml
grades_one:                           # source reliability
  default: Usually reliable           # optional — names must exist in items
  items:
    - Always reliable
    - Usually reliable
    - Fairly reliable
    - Not usually reliable
    - Unreliable

grades_two:                           # information reliability
  items:
    - Confirmed
    - Probably true
    - Possibly true

grades_three:                         # third dimension
  items: [High, Medium, Low]
```

`grade_one`/`grade_two`/`grade_three` on entities, links, and cards
accept **either** the 0-based index into `items` **or** the registered
grade name string. So `grade_one: 0` and `grade_one: 'Always reliable'`
produce the same XML. Unknown names raise `unknown_grade`; there is no
auto-create.

> **Important shape note.** Grades take a dict with `items` (and optional
> `default`) — **not** a bare list. `grades_one: ['A', 'B']` is silently
> dropped by the loader. Use `grades_one: {items: [A, B]}`.

Full details: [cards-and-grades.md](../reference/cards-and-grades.md).

### Source types

A flat list of free-text labels:

```yaml
source_types:
  - Witness
  - Informant
  - Officer
  - Intelligence
```

Reference them on entities, links, and cards via `source_type: Witness`.

### Strengths

```yaml
strengths:
  default: Confirmed                  # optional
  items:
    - name: Confirmed
      dot_style: solid
    - name: Probable
      dot_style: dashed
    - name: Possible
      dot_style: dotted
```

Same shape constraint as grades: it's a `{items, default?}` dict, not a bare
list. `dot_style` values: `solid`, `dashed`, `dash_dot`, `dash_dot_dot`,
`dotted` (or aliases `-`, `---`, `-.`, `-..`, `...`).

Full details: [visual-styling.md § Strengths](../reference/visual-styling.md#strengths).

## Settings

`settings` is a nested dict mirroring the `Settings` dataclass groups. Every
field is optional. Set only what you want to override; ANB picks its own
defaults for everything else.

```yaml
settings:
  chart:
    bg_color: 16777215                # white (COLORREF int, name, or '#RRGGBB')
    rigorous: true

  view:
    time_bar: true                    # show ANB's time bar
    show_pages_boundaries: false

  grid:
    snap: true
    visible: false

  extra_cfg:                          # anxwritter-only knobs (NOT in ANX XML)
    arrange: grid                     # radial (default) | circle | grid | random | fr | forceatlas2 | tree
    layout_scale: 1.5                 # uniform spread multiplier across all arrange modes (default 1.0)
    entity_auto_color: true
    link_match_entity_color: true
    link_arc_offset: 25

  legend_cfg:
    show: true
    x: 50
    y: 50
    arrange: wide                     # wide | tall | square
    font:
      name: 'Segoe UI'
      size: 10
      bold: true

  summary:
    title: 'Operation Sunshine'
    author: 'J. Doe'
    keywords: 'fraud, money'
    custom_properties:
      - name: Case
        value: 'CR-2024-001'
      - name: Classification
        value: 'RESTRICTED'

  font:                               # chart-wide default font
    name: Tahoma
    size: 10
```

Full per-group field reference: [settings.md](../reference/settings.md).
Every field listed there is settable from YAML/JSON using the same name.

### Geo-map

`extra_cfg.geo_map` maps an entity attribute value to lat/lon coordinates
for canvas positioning and/or ANB's Esri Maps integration.

```yaml
settings:
  extra_cfg:
    geo_map:
      attribute_name: City            # match this entity attribute
      mode: both                      # 'position' | 'latlon' | 'both'
      width: 3000
      height: 2000
      spread_radius: 30
      data:
        New York: [40.7128, -74.0060]
        London: [51.5074, -0.1278]
        Tokyo: [35.6762, 139.6503]

entities:
  icons:
    - id: alice
      type: Person
      attributes:
        City: 'New York'              # matched against geo_map.data keys
```

Or load coordinates from an external file:

```yaml
settings:
  extra_cfg:
    geo_map:
      attribute_name: City
      mode: both
      data_file: city_coords.json     # JSON or YAML, key -> [lat, lon]
```

The external file is a flat top-level mapping of key → `[lat, lon]` — same
shape as the inline `data:` block above. Both formats are accepted; pick by
file extension.

`city_coords.json`:

```json
{
  "New York": [40.7128, -74.0060],
  "London": [51.5074, -0.1278],
  "Tokyo": [35.6762, 139.6503]
}
```

`city_coords.yaml`:

```yaml
New York: [40.7128, -74.0060]
London: [51.5074, -0.1278]
Tokyo: [35.6762, 139.6503]
```

The CLI's `--geo-data` flag does the same thing on the command line:

```bash
anxwritter --config org.yaml --geo-data city_coords.json data.yaml -o out.anx
```

Matching is **case- and accent-insensitive by default**: `São Paulo/SP`,
`SAO PAULO/SP`, and `sao paulo/sp` all match the same key. Set
`accent_insensitive: false` under `geo_map` if you need strict matching
(e.g. when accented and unaccented forms intentionally point at different
coordinates).

Full GeoMapCfg field details:
[settings.md § GeoMapCfg](../reference/settings.md#geomapcfg-fields).

## Legend items

Each item is a row in the chart legend. Requires `legend_cfg.show: true` to
make the legend panel visible.

```yaml
legend_cfg:
  show: true

legend_items:
  - name: Person                      # NOT 'label' — see gotchas
    item_type: Icon
    image_name: adult                 # ANB icon key
  - name: Calls
    item_type: Link
    color: 16711680
    line_width: 1
    arrows: '->'
  - name: Confirmed
    item_type: Line
    color: 0
    line_width: 2
    dash_style: solid
  - name: Section header
    item_type: Font
    font:
      bold: true
      size: 11
```

`item_type` accepts lowercase (`'icon'`, `'link'`) or legacy Title Case
(`'Icon'`, `'Link'`). For backward compat the loader also accepts `label:`
as an alias for `name:` on legend items, but `name:` is the canonical key.

ANB always groups legend items by type in a fixed order, regardless of how
you list them: **Text → Attribute → Icon → Line → Font → Link → TimeZone →
IconFrame**.

Full LegendItem field reference + behaviour notes:
[visual-styling.md § Legend items](../reference/visual-styling.md#legend-items).

## Palettes

Optional. When omitted, anxwritter auto-generates a single `"anxwritter"`
palette containing all registered entity types, link types, and attribute
classes. Define palettes explicitly to control the "Insert from Palette"
panel:

```yaml
palettes:
  - name: Investigation
    locked: false
    entity_types: [Person, Company, Phone]
    link_types: [Calls, Transfers Money]
    attribute_classes: [Phone Number, Balance]
    attribute_entries:
      - name: Phone Number
        value: '(555) 0100-0000'
```

Full Palette field reference:
[visual-styling.md § Palettes](../reference/visual-styling.md#palettes).

## Semantic types

Use these only if your organisation works with i2's Semantic Type Library.
For most charts you can skip this section — entity types and link types
already drive icons and colors.

```yaml
semantic_entities:
  - name: Pessoa                      # custom type extending the Entity root
    kind_of: Entity
  - name: Suspeito
    kind_of: Pessoa

semantic_links:
  - name: Surveilled
    kind_of: Link

semantic_properties:
  - name: CPF
    base_property: Abstract Text      # or Abstract Number / Abstract Date & Time / Abstract Flag

# Reference custom semantic types from entity/link/attribute definitions
entity_types:
  - name: Suspeito
    icon_file: adult
    semantic_type: Suspeito           # by name (resolved to GUID) or raw 'guid…'

link_types:
  - name: Surveilled
    color: 255
    semantic_type: Surveilled

attribute_classes:
  - name: CPF
    type: text
    semantic_type: CPF
```

Per-instance overrides for entities and links (not for attribute classes):

```yaml
entities:
  icons:
    - id: alice
      type: Person
      semantic_type: Suspeito         # overrides the EntityType-level setting

links:
  - from_id: alice
    to_id: bob
    type: Call
    semantic_type: Surveilled
```

Full semantic-type rules (resolution order, catalogue emission, the LN-prefix
warning): [semantic-types.md](../reference/semantic-types.md).

## YAML gotchas

Things that look right but produce wrong output if you don't know about them.

### 1. Quote times (dates are fine either way)

```yaml
# Wrong — YAML parses unquoted `time:` as an integer
time: 14:30:00                        # → integer 52200 (FAILS validation)

# Right
time: '14:30:00'                      # → string "14:30:00"
```

Unquoted `time: 14:30:00` is YAML's sexagesimal-integer notation — `14*3600 +
30*60 + 0 = 52200`. Validation catches it as `invalid_time` (including for
inline cards), but quoting is the simpler fix.

Unquoted `date: 2024-01-15` is fine — YAML parses it as a `datetime.date`,
which the loader now accepts directly. Same for unquoted ISO datetimes in
`settings.summary.created`, `settings.time.default_datetime`, and any
attribute value (where the loader emits an `AttTime` attribute):

```yaml
# All of these work
date: 2024-01-15                      # → date object → emitted as 2024-01-15
settings:
  summary:
    created: 2024-01-15T10:00:00      # → datetime → emitted as 2024-01-15T10:00:00
attributes:
  Joined: 1990-05-12                  # → date → emitted as AttTime 1990-05-12T00:00:00
```

### 2. YAML 1.1 boolean sentinels

YAML 1.1 (which `pyyaml` uses by default via `safe_load`) treats more than
just `true`/`false` as booleans:

```yaml
attributes:
  Active: yes                         # → Python bool True (Flag attribute)
  Married: no                         # → Python bool False (Flag attribute)
  On: on                              # → bool True
  Off: off                            # → bool False
```

If you mean a string, quote it:

```yaml
attributes:
  Status: 'yes'                       # → string "yes" (Text attribute)
```

This is YAML's behaviour, not anxwritter's — JSON has none of these
ambiguities.

### 3. Quote arrow shorthand

```yaml
arrow: ->                             # parse error: '>' is YAML-special
arrow: '->'                           # right
arrow: head                           # also right (no quoting needed)
```

Same for `<-` and `<->`.

### 4. Enums are lowercase strings

The Python API accepts `DotStyle.SOLID`, `MergeBehaviour.ADD`,
`LegendItemType.ICON`, etc. YAML/JSON has no way to reference Python enums,
so use the lowercase string value:

```yaml
strengths:
  items:
    - name: Confirmed
      dot_style: solid                # not DotStyle.SOLID

attribute_classes:
  - name: Notes
    type: text
    merge_behaviour: add_line_break   # not MergeBehaviour.ADD_WITH_LINE_BREAK

legend_items:
  - name: Person
    item_type: icon                   # not LegendItemType.ICON
```

The lowercase values are listed in each enum's reference table — see
[visual-styling.md](../reference/visual-styling.md) and
[attributes.md](../reference/attributes.md).

### 5. Color forms

```yaml
color: Blue                           # named color (case-insensitive)
color: blue                           # equivalent
color: 'Light Orange'                 # quote multi-word names
color: light_orange                   # snake_case also works
color: '#FF6600'                      # hex (quote — '#' starts a YAML comment otherwise)
color: 16711680                       # COLORREF int (R + G*256 + B*65536)
```

Full list of 40 named colors and their RGB values:
[visual-styling.md § Named colors](../reference/visual-styling.md#all-40-named-colors).

### 6. `name:`, not `label:` on legend items

The dataclass field is `name`. Older YAML configs sometimes used `label:`
and the loader still accepts it for back-compat, but `name:` is the canonical
key going forward. The text becomes the legend row's display label.

### 7. Bare lists for grades / strengths are silently dropped

```yaml
# Wrong — silently produces an empty grades_one collection
grades_one:
  - Always reliable
  - Usually reliable

# Right
grades_one:
  items:
    - Always reliable
    - Usually reliable
```

Same applies to `grades_two`, `grades_three`, and `strengths`.

### 8. Datetime format names — never inline strings

```yaml
# Wrong — ANB rejects inline format strings with "NULL argument passed to SetDateTimeFormat"
entities:
  icons:
    - id: A
      type: P
      datetime_format: 'yyyy-MM-dd'

# Right — register a named format, then reference by name
datetime_formats:
  - name: ISO
    format: 'yyyy-MM-dd'

entities:
  icons:
    - id: A
      type: P
      datetime_format: ISO
```

Validation catches this with the `unregistered_datetime_format` error.

### 9. `timezone` needs both `date` AND `time`

```yaml
# Silently ignored by ANB (validation warns with timezone_without_datetime)
- id: A
  type: P
  date: '2024-01-15'
  timezone:
    id: 1
    name: UTC
  # missing time:

# Right
- id: A
  type: P
  date: '2024-01-15'
  time: '00:00:00'
  timezone:
    id: 1
    name: UTC
```

### 10. `ordered: true` on a link only works between two ThemeLines

ANB rejects `ordered=True` if either end isn't a ThemeLine; validation
catches this as `invalid_ordered`. The constraint doesn't apply to entities.

## JSON-specific notes

JSON is the default for stdin (`cat input.json | anxwritter -o …`) and is
detected from the `.json` extension on file inputs.

JSON has no native datetime, no `yes`/`no` sentinels, no `'->'` quoting
issues — most of the YAML gotchas above don't apply. The only thing to be
aware of:

### Datetime attribute values

JSON has no datetime literal, so a value like `"Joined": "2024-01-15T10:00:00"`
parses as a plain Python string. To get a DateTime-typed attribute (rendered
as `AttTime` in the ANX XML, formatted by ANB as a date/time field rather
than a text field), **declare the attribute name in `attribute_classes` with
`type: datetime`**:

```json
{
  "attribute_classes": [
    {"name": "Joined", "type": "datetime"}
  ],
  "entities": {
    "icons": [{
      "id": "alice", "type": "Person",
      "attributes": {"Joined": "2024-01-15T10:00:00"}
    }]
  }
}
```

The loader sees the declaration and parses the string into a `datetime`
before the builder runs. Accepted forms:

| Format | Example |
|--------|---------|
| ISO 8601 datetime | `"2024-01-15T10:00:00"` |
| ISO 8601 with microseconds | `"2024-01-15T10:00:00.500000"` |
| ISO 8601 with `Z` (UTC marker) | `"2024-01-15T10:00:00Z"` |
| ISO 8601 with space separator | `"2024-01-15 10:00:00"` |
| ISO 8601 date-only | `"2024-01-15"` |
| `dd/MM/yyyy` | `"15/01/2024"` |
| `yyyymmdd` | `"20240115"` |

A declared-datetime attribute whose value can't be parsed (e.g.
`"Joined": "unknown"`) is left as a string and the validator surfaces a
`type_conflict` error so you get a clear, actionable signal.

If you don't declare the attribute in `attribute_classes`, the value
remains a string (Text attribute) — there is no auto-detection. This is
deliberate: opaque IDs like `"OrderID": "2024-01-15-A001"` would otherwise
get wrongly typed as DateTime.

Otherwise the JSON shape is identical to the YAML shape. Every example on
this page can be transliterated 1-to-1.

## Config files

A **config file** holds only the configuration sections (`settings`,
`entity_types`, `link_types`, `attribute_classes`, `strengths`,
`datetime_formats`, `palettes`, grades, `source_types`, `legend_items`,
semantic types). It deliberately omits `entities` and `links`, so you can
share one config across many investigation files.

```bash
# One config + one data file
anxwritter --config org.yaml data.yaml -o out.anx

# Layered configs — applied in order, "later wins per name"
anxwritter --config base.yaml --config team_overrides.yaml data.yaml -o out.anx

# Config-only (produces an empty chart with all the type definitions)
anxwritter --config org.yaml -o template.anx
```

When a config is loaded, the **data file cannot redefine** a config-locked
entity type, link type, attribute class, strength, datetime format, grade
collection, or source type list with different specs. Identical specs are
silently skipped; conflicting specs produce a `config_conflict` validation
error.

For full layering rules and conflict detection: see
[constructors.md § Config files](../reference/constructors.md#config-files).

## CLI quick reference

```bash
# JSON or YAML input → .anx
anxwritter chart.yaml -o output/chart.anx
anxwritter chart.json -o output/chart.anx

# Stdin (JSON only)
cat chart.json | anxwritter -o output/chart.anx

# Validate only — prints errors to stderr as JSON, exits 0 on success
anxwritter chart.yaml --validate-only

# With one or more config files (layered, applied in order)
anxwritter --config org.yaml --config overrides.yaml data.yaml -o output/chart.anx

# Inspect the resolved merged config (with provenance comments)
anxwritter --show-config --config org.yaml --config team.yaml

# External geo coordinate file
anxwritter --config org.yaml --geo-data coords.json data.yaml -o output/chart.anx
```

## Loading from Python

You don't have to use the CLI — you can load a YAML/JSON file from Python:

```python
from anxwritter import ANXChart

# YAML (requires pyyaml: pip install anxwritter[yaml])
chart = ANXChart.from_yaml_file('chart.yaml')

# JSON
chart = ANXChart.from_json_file('chart.json')

# Or from a string
chart = ANXChart.from_yaml(yaml_text)
chart = ANXChart.from_json(json_text)

# Or from an already-parsed dict
chart = ANXChart.from_dict(data_dict)

# With a config layered on top
chart = ANXChart(config_file='org.yaml')
chart.apply_config_file('overrides.yaml')   # optional second layer
data = yaml.safe_load(open('data.yaml'))
chart._apply_data(data)                     # internal — public-API equivalent: from_dict on the merged data

chart.to_anx('output/chart.anx')
```

All four `from_*` constructors are documented in
[constructors.md](../reference/constructors.md).

## Validation

`anxwritter --validate-only` runs the same validation that
`anxwritter to_anx` runs internally, but doesn't write a file. Output is a
JSON array of error dicts on stderr (or `[]` on stdout for success):

```json
[
  {
    "type": "missing_required",
    "message": "Missing required field 'type' on entity 'alice'",
    "location": "entities[0] (Icon)"
  },
  {
    "type": "type_conflict",
    "message": "Attribute 'Balance' used as 'Text' but previously seen as 'Number'",
    "location": "entities[3] (Icon)"
  }
]
```

Every error dict has `type`, `message`, and (almost always) `location`. The
`config_conflict` error additionally includes `section`, `name`,
`config_value`, and `data_value`.

Full list of error types and their triggers:
[validation.md](../reference/validation.md).
